<objective>
Build the main-process side of terminal session reconnect: a stable `viewId → {termId, outputBuf}` registry in `electron/ipc/terminal.ts`, a viewId-aware `terminal:create`, a new `terminal:reconnect` handler that returns a live session (or null), buffer cleanup in `terminal:kill`, plus the preload and typed-client plumbing that exposes `terminalReconnect` to the renderer. The 50 KB output-buffer cap is extracted into a pure `src/shell/termBuffer.ts` helper so the capping logic is unit-testable (Vitest only includes `src/**`, never `electron/**`).

Purpose: This is the contract the renderer consumes in plan 05-02. Without the main-process registry surviving a renderer reload and the `terminalReconnect` IPC channel existing, the renderer has nothing to reconnect to. This plan delivers TERM-01 (registry + viewId-keyed create), TERM-02 (reconnect handler returning the live session and re-wiring the sender), and TERM-04 (50 KB cap + orphaned-pty cleanup on dead reconnect + dual-map kill cleanup).

Output:
- `src/shell/termBuffer.ts` — pure `appendCapped(buf, chunk)` + `TERM_BUFFER_CAP = 50000`
- `src/shell/__tests__/termBuffer.test.ts` — unit tests for the cap behavior
- `electron/ipc/terminal.ts` — `viewSessions` registry, viewId-aware `terminal:create`, `terminal:reconnect`, kill cleanup
- `electron/preload.ts` — `terminalReconnect(viewId)` bridge method
- `src/ipc/client.ts` — typed `terminalReconnect(viewId)` wrapper
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer → main (IPC) | Renderer supplies viewId and (on create) cwd as untrusted IPC arguments |
| pty → main → renderer | Live shell output crosses the main process into a renderer window's data channel |
| renderer window lifetime | A Vite full page reload destroys the renderer window; the original event.sender becomes stale |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information disclosure | a viewId from window A reconnecting to window B's shell output | accept | viewId is the app's own stable ViewInstance id, generated and held entirely within the single trusted renderer; there is one user and one renderer process — no cross-user boundary exists. Output is the user's own shell. |
| T-05-02 | Denial of service | unbounded outputBuf growth from a long-running noisy shell | mitigate | appendCapped enforces the 50 KB TERM_BUFFER_CAP on every accumulation site (create onData + reconnect onData); buffer can never exceed 50000 chars |
| T-05-03 | Tampering | stale event.sender from the pre-reload window still wired to onData, sending to a destroyed window | mitigate | every send is guarded by `!event.sender.isDestroyed()`; reconnect registers a fresh onData/onExit bound to the new window's sender; the stale listener becomes a guarded no-op |
| T-05-04 | Resource leak (DoS) | orphaned viewSessions entries pointing at exited ptys accumulating across reloads | mitigate | reconnect deletes the entry when `!processes.has(entry.termId)`; terminal:kill removes entries by termId from both maps; disposeTerminals clears the registry |
| T-05-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages are installed in this phase (node-pty, electron, @xterm/* already present) — slopcheck N/A; no install tasks exist |
</threat_model>
