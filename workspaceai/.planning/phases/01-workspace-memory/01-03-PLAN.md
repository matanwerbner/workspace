<objective>
Wire the renderer to the memory storage layer: expose the `memory:*` IPC channels through the preload bridge and typed client, add the per-tool `alwaysAllow` flag to `AiTool`, define the two global memory tools, and patch `ChatPanel.tsx` to (1) make both tools available in every view type, (2) inject the MEMORY.md index into the system prompt, (3) auto-execute memory tools with no approval prompt, and (4) dispatch `read_memory` / `write_memory` to the IPC layer.

Purpose: This is the user-facing payoff. After this plan, Claude in ANY view can read the injected memory index, fetch entries on demand, and proactively save new memory — all without approval friction — scoped to the active workspace's homeFolder.

Output:
- `src/views/types.ts` — `AiTool.alwaysAllow?: boolean`
- `electron/preload.ts` + `src/ipc/client.ts` — three `memory*` bridge/wrapper methods
- `src/shell/memory.ts` — `GLOBAL_MEMORY_TOOLS` definition (read_memory + write_memory, both `alwaysAllow: true`) and a `shouldBypassApproval` predicate
- `src/shell/__tests__/memoryTools.test.ts` — tests for tool shapes, alwaysAllow, and the bypass predicate
- `src/shell/ChatPanel.tsx` — global-tool merge, index injection, per-tool approval bypass, memory tool dispatch
</objective>

| T-01-10 | Tampering | model supplies a malicious name/topic (e.g. '../secrets') to write_memory/read_memory | mitigate | isValidMemoryName in the main-process handler (plan 01-01) rejects it before any fs op; renderer passes args straight through but the server-side gate is authoritative |
| T-01-11 | Information disclosure | memory index leaking into prompts of an unrelated workspace | mitigate | index is fetched with selectActiveWorkspace().homeFolder; memory is per-workspace by construction and skipped entirely when homeFolder is unset |
